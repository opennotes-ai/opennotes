# OpenNotes Documentation

The source for [docs.opennotes.ai](https://docs.opennotes.ai), built with [Mintlify](https://mintlify.com).

## Prerequisites

- Node 22+ and pnpm
- Mintlify CLI: `npm i -g mint` (or use `pnpm dlx mint`)
- `mise` (optional, provides the `docs:*` task shortcuts)

## Local development

From this directory:

```bash
pnpm install            # no-op; we have no runtime deps, just scripts
mise run docs:dev       # preferred — vendors @opennotes/tokens first, then runs `mint dev`
```

Or without mise:

```bash
pnpm run dev            # equivalent — runs vendor:tokens then mint dev
```

Mintlify serves the site at http://localhost:3000 and live-reloads on MDX changes.

## Project layout

```
opennotes-docs/
  docs.json              # Mintlify config — theme, colors, top-level tabs, nav groups
  style.css              # Auto-loaded by Mintlify; @imports vendored @opennotes/tokens
  introduction.mdx       # Landing page (docs.opennotes.ai/)
  integration-guide/     # Universal adapter contract (onboarding, concepts, choreography)
  existing-integrations/ # Shipped adapters (Discourse today; Discord later)
  api-reference/         # Generated from ../opennotes-server/openapi-public.json
  snippets/              # Reusable MDX fragments (e.g., preview-callout)
  styles/tokens/         # GITIGNORED — populated by vendor:tokens from ../packages/tokens/
  INTEGRATION.md         # Authoritative spec for @opennotes/tokens vendoring
```

- **Tabs** live in `docs.json > navigation.tabs`. Three tabs: Integration Guide, Existing Integrations, API Reference.
- **New pages** must be added to the appropriate group in `docs.json` or they won't appear in the sidebar.
- **File names** become URL slugs. Slug `integration-guide/overview` renders at `/integration-guide/overview`.

## Content conventions

- MDX only (`.mdx`), not plain markdown. Lets us embed Mintlify components like `<Note>`, `<Card>`, `<Steps>`.
- Code fences should specify a language for syntax highlighting (`bash`, `json`, `python`, `typescript`, `ruby`).
- Cross-link with relative slugs: `[authentication](/integration-guide/concepts/authentication)`.
- Link to platform/server URLs with absolute URLs: `https://platform.opennotes.ai`.

## Regenerating the API reference

The API reference is rendered from `../opennotes-server/openapi-public.json`. The file is committed into the `opennotes-server` tree and regenerated by the server's own schema workflow.

To refresh it after server route changes:

```bash
mise run docs:regenerate-openapi
```

(That delegates to the `opennotes-server` mise task `docs:generate:openapi`.)

A drift check (`opennotes-server/scripts/check_openapi_sync.py`) runs in CI on the server repo and blocks merges that leave `openapi-public.json` out of sync with the live FastAPI schema.

## Vendored artifacts

Mintlify only reads files under the docs project root. Two external artifacts are vendored in before every build:

| Artifact | Source | Vendored path | Script |
|---|---|---|---|
| `@opennotes/tokens` CSS | `../packages/tokens/src/*.css` | `styles/tokens/*.css` | `vendor:tokens` |
| OpenAPI schema | `../opennotes-server/openapi-public.json` | `./openapi-public.json` | `vendor:openapi` |

Both are gitignored locally and regenerated by `pnpm run vendor:all` (which `predev`, `prevalidate`, and `prebroken-links` invoke automatically). The design-tokens integration is spec'd in [INTEGRATION.md](./INTEGRATION.md).

After cloning you must run `pnpm run vendor:all` (or `pnpm run dev`/`pnpm run validate`/`pnpm run broken-links`, which all do it for you) at least once for theme variables to resolve and the API Reference tab to render.

## Deployment

`docs.opennotes.ai` is hosted by Mintlify Cloud with the **content root** set to `opennotes/opennotes-docs/` (so Mintlify ignores the rest of the opennotes monorepo). Preview deploys fire on PRs to the `opennotes-ai/opennotes` repo that touch this directory.

DNS (a `docs` CNAME → Mintlify's publishing host) is configured in the `infrastructure` repo (`environments/production/dns.tf`). The deploy runbook is tracked in TASK-1460.06.

## Adding content

1. Create or edit an `.mdx` file under the relevant section.
2. If it's a new file, add its slug to `docs.json > navigation.tabs[...].groups[...].pages`.
3. Run `mise run docs:dev` and navigate to the page.
4. Before committing: `mise run docs:broken-links` to catch stale cross-links.

## Continuous integration

`.github/workflows/docs-check.yml` runs on every PR that touches `opennotes-docs/**` or `packages/tokens/src/**`. The workflow:

1. Vendors `@opennotes/tokens` CSS + `openapi-public.json` into the docs project root (same artifacts `pnpm run vendor:all` produces locally).
2. Validates that `docs.json` is parseable JSON with the required top-level keys.
3. Runs `mint broken-links` — fails the job on any broken internal link.
4. Runs `mint validate` — strict build; fails on docs.json schema errors, missing pages referenced from nav, or OpenAPI-source resolution errors.

Reproduce locally:

```bash
mise run docs:broken-links     # link check
mise run docs:validate          # strict build
```

OpenAPI drift is gated separately by the `validate-api-endpoints` job in `.github/workflows/ci.yml`, which runs `scripts/check_openapi_sync.py`. That blocks any server PR whose code changes aren't reflected in the committed `openapi.json` / `openapi-public.json`.
